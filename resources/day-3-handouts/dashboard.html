Quick lookup for terminal, git, Claude, and setup
This is a starting point, not a fixed document. It covers the terminal commands, git workflows, Claude setup, and troubleshooting you're most likely to need — but tools change and your needs will differ. You can give this file to Claude and ask it to add sections, update commands, restyle it, or build you a completely different version from scratch. That's part of the point: learning to make your own resources is as valuable as the resources themselves.

Navigate and inspect

pwd                        # where am I?
ls                         # what's here?
ls -la                     # hidden files, sizes, permissions
cd directory-name          # move into a directory
cd ..                      # up one level
cd ~                       # home directory

Read files

cat filename               # print file contents
cat -n filename            # print with line numbers
head -20 filename          # first 20 lines
cat -n file | head -20     # first 20 lines with line numbers
Pager: if output fills the screen and you're stuck, press q to exit. Ctrl+C cancels a running command.

Find files

find . -name "*.py"            # all Python files from here down
find . -name "CLAUDE.md"       # specific filename
Tip: find searches everything including node_modules/ and .next/. Those are dependency files, not files you wrote.

Search inside files (grep)

grep -r "search term" .                  # recursive search
grep -rn "search term" .                 # with line numbers
grep -ri "search term" .                 # case-insensitive
grep -rn "def " . --include="*.py"       # only Python files

Create and manage files

touch filename.txt             # create an empty file
mkdir directory-name           # create a directory
mkdir -p path/to/nested/dir    # create nested directories
mv old-name new-name           # rename or move
cp source destination          # copy
rm filename                    # delete a file (no undo!)
rm -r directory-name           # delete a directory and contents

If you lose the cursor

Symptoms: typing does nothing, the prompt disappeared, or you see something weird like dquote> or > or :.
What you seeWhat happenedFix
dquote> or quote>Unclosed quote — terminal is waiting for the closing " or 'Ctrl+C to cancel
> at the start of a lineUnclosed command (backslash, pipe, etc.)Ctrl+C to cancel
Colon : at the bottom of the screenYou're in the pager (less) — git log, man pages, long outputPress q to exit
Nothing happens when you typeA process is running in the foregroundCtrl+C to cancel it
Screen looks garbled or frozenOutput messed up the displayType reset and press Enter (even if you can't see it)
(END) at the bottomPager showing the end of outputPress q
The universal escape hatch: Ctrl+C cancels almost anything. If that doesn't work, Ctrl+D closes the shell session entirely. Worst case: close the terminal window and open a new one.

Other shortcuts

q                              # exit the pager (git log, man pages)
Ctrl+C                         # cancel / exit running process
Ctrl+D                         # end input / exit shell session

Day-to-day

git status                         # what's changed?
git diff                           # unstaged changes (working vs staging)
git diff --staged                  # staged changes (staging vs last commit)
git add filename                   # stage a specific file
git commit -m "message"            # commit with message
git log --oneline                  # compact history
git log --oneline --graph          # history with branch visualization
git restore filename               # discard working changes
git restore --staged filename      # unstage a file
The two diffs: git diff shows what's changed but not staged. git diff --staged shows what's staged but not committed. Two sides of the same coin — git add moves changes from one to the other.

Branches

git branch                         # list local branches
git switch branch-name             # switch to a branch
git switch -c branch-name          # create and switch in one step
git merge branch-name              # merge into current branch
git branch -d branch-name          # delete a branch (safe)
git push origin --delete branch    # delete a remote branch

Push and pull

git push                           # push to tracked remote
git push -u origin branch-name     # push new branch + set tracking (-u = "remember this link")
git pull                           # fetch and merge from remote
git pull origin branch-name        # pull a specific branch into current
What -u does: short for --set-upstream. It tells git "from now on, when I say git push or git pull on this branch, I mean this remote." You only need it the first time you push a new branch. After that, bare git push and git pull just work.
The reflex: pull before you push. If the remote has commits you don't have, git rejects the push.

Pull requests (terminal)

# Full PR workflow
git switch -c my-feature
# ... make changes ...
git add .
git commit -m "describe changes"
git push -u origin my-feature
gh pr create --title "Title" --body "Description"

# Merge a PR
gh pr merge 12

# After merge, clean up
git switch main
git pull
git branch -d my-feature
Why "pull request"? The name is from the reviewer's perspective. You're requesting that they pull your changes into their branch. You push (logistics), then open a PR (social act).

New repo from a local project

The clean path (with gh CLI)

# 1. Make sure work is committed
git add -A
git commit -m "describe the state you're saving"

# 2. Create empty repo, wire up origin, push — all at once
gh repo create my-project --private --source=. --remote=origin --push
FlagWhat it does
--private / --publicVisibility
--source=.Use current directory's git history
--remote=originName the remote "origin"
--pushPush immediately after creating
The snafu: if you create the repo on GitHub's website and check "Add README" or "Add .gitignore," GitHub makes an initial commit. Your local repo has never seen it. The push will fail with "diverged histories." Fix: always create the remote empty.

Push to an organization

gh repo create org-name/repo-name --private --source=. --push

The website fallback (no gh CLI)

# On GitHub.com: New repo, leave ALL checkboxes unchecked
git remote add origin https://github.com/USERNAME/repo.git
git push -u origin main

Pre-push checklist

git log --oneline          # at least one commit exists?
git branch                 # branch named main?
git remote -v              # origin points to right repo?

Create PR from Claude Code desktop

  1. Be on a branch (not main) with commits
  2. The Create PR tab in the top-right becomes active
  3. Click it — it pushes the branch and opens a PR form
  4. Write a title and description, create
  5. Review and merge on GitHub
  6. After merge: git switch main && git pull
If the Create PR tab is missing: check git log main..HEAD --oneline — if it's empty, there's nothing to PR (you're on main or have no commits ahead).

Three interfaces: Chat / Cowork / Code

Chat — ask Claude something, get an answer. Cowork — give Claude access to your files, work on them together. Code — build workflows that run on your files automatically, repeatedly, at scale.

Cowork (desktop app)Working on a folder of files via GUIReads/edits real files, but how stays hidden
Claude Code (terminal/IDE)Full visibility, multi-file projects, automationEvery file read, command run, and edit is visible

When to upgrade

Chat → Cowork when you're pasting the same document repeatedly, want Claude to read files directly, or the task spans multiple sessions.

Cowork → Code when you're doing the same task more than a few times, need to operate on many files at once, want to connect external services, or want to build something a colleague can use.

The honest tradeoff: more powerful interfaces require more setup. Chat is available in 30 seconds. Cowork takes an afternoon. Code takes longer and rewards sustained investment. The right interface is the one you'll actually use.

CLAUDE.md

A plain-text file in a project's root that gives Claude persistent instructions. Claude reads it automatically when working in that folder.

Good things to put in it

  • Your role and the audience you write for
  • Conventions for file naming, formatting, style
  • Things you've corrected Claude on more than twice
  • "Always ask before deleting files"
  • Project-specific context (what the inputs are, what the goal is)

Claude Code setup (Mac)

# Install Claude Code
curl -fsSL https://claude.ai/install.sh | bash

# Close and reopen terminal, then verify
claude --version

# Health check — verifies CLAUDE.md, settings, hooks, and MCP servers are loading correctly
claude doctor

# Start Claude Code (from inside a project folder)
claude

# If "command not found":
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc

  1. Open Extensions (Cmd+Shift+X), search "Claude Code"
  2. Install the one published by Anthropic (check publisher)
  3. Click the spark icon in the Activity Bar to open the Claude panel
  4. Sign in when prompted

Cowork setup

  1. Open Claude desktop app, confirm you see your plan indicator (bottom-left)
  2. Click the Cowork tab
  3. Create a new project → name it → dock a folder to it
  4. Test: "Can you list the files in this folder?"
One folder per project. Scope each project to the smallest folder that holds what it needs. Start a new task inside the same project (instead of a new project) when context rots in a long thread.

Permissions: Allow Once vs Always

Claude Code asks permission before reading files, running commands, or editing code. Each prompt gives you two options:

OptionWhat it meansHow long it lasts
Allow OnceApprove this specific action right nowThis one action only — Claude asks again next time
Always AllowApprove this type of action for this projectPersists across sessions — saved to your project settings file
"Always" means always (for this project). It doesn't reset at the end of a session. It's saved in .claude/settings.json in the project folder. To revoke an "Always" permission, edit that file — or just ask Claude to do it ("remove the always-allow for X").
When to use Always: common read-only actions you'll do every session (reading files, running git status, running tests). When to stick with Once: anything that modifies files, installs packages, or runs unfamiliar commands — at least until you understand what it's doing.

Tips and gotchas

  • "Command not recognized" right after installing → quit terminal fully, open fresh window
  • Context rot: long conversations drift. Use /compact or start a new task in the same project
  • Check what changed: when Claude edits files, glance at the diff before moving on
  • @ to attach files: in the Claude panel prompt, type @ to fuzzy-search project files
  • Type /context to see what's filling your context window

The sequence (in order)

Each step depends on the one before it. Don't skip ahead.

1. Open Terminal

Cmd+Space → type "Terminal" → Enter. They may have never opened it before.

2. Install Command Line Tools

xcode-select --install

Click "Install" in the dialog. Wait 5-15 minutes. If it says "already installed," skip to step 3.

3. Install Homebrew

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
  • Password prompt shows nothing as they type — that's normal
  • Don't let them scroll past the output! It prints PATH commands they need to run

The PATH commands look like:

echo >> /Users/username/.zprofile
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> /Users/username/.zprofile
eval "$(/opt/homebrew/bin/brew shellenv)"
They must copy their own output, not a generic version. The path includes their username.

4. Verify Homebrew

brew --version

If "command not found": the PATH step was missed. Go back to step 3.

5. Install core tools

brew install git
brew install node
brew install --cask visual-studio-code
brew install python@3.13
After all four finish: Cmd+Q Terminal, open a fresh window. New installs aren't visible to an already-open terminal.

6. Verify everything

git --version
node --version
npm --version
python3 --version
code --version

Install packages

JavaScript / Node

pnpm install                   # install all from package.json
pnpm add package-name          # add a new package
pnpm add -D package-name       # add as dev dependency
npm install                    # same thing, different manager

Python

pip install package-name
pip install -r requirements.txt    # install from requirements file
pip freeze > requirements.txt      # save environment to file
What's a package? A package is code someone else wrote that you can add to your project. package-name is a placeholder — you replace it with the actual name of what you need (e.g., pip install requests or pip install pandas). You'll usually find the name in a project's README, a tutorial, or an error message that says something is missing.

Start a project

Next.js app

pnpm create next-app@latest directory-name

Scaffolds a full project with prompts for TypeScript, Tailwind, etc. Don't mkdir first.

Python project (with virtual environment)

mkdir project-name
cd project-name
python3 -m venv .venv
source .venv/bin/activate      # prompt changes to (.venv)

Plain folder + git

mkdir project-name
cd project-name
git init

Run a project

Next.js / Node

pnpm dev          # start dev server (usually localhost:3000)
pnpm build        # production build
pnpm start        # run the production build

Python

python3 script.py
python3 -m module_name    # run a module (e.g., python3 -m pytest)
